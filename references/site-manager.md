# Site Manager (site-builder-phase2)

The engine behind BGD's productized tier. A chat-driven, multi-tenant site builder that orchestrates Claude Code headless against client site repos. The Terraplex platform is one configuration of this engine — it's not Terraplex-specific.

## Core mechanic

```
Browser chat UI (Vue 3) → SSE → Express backend → spawns `claude --print` headless
                                                  pointed at sites/<slug>/
                                                  with allowed tools: Edit, Write, Bash, Read
                              ↓
              Claude edits files in the site repo, output streams back to browser
                              ↓
                      Backend runs `git diff`, returns to UI
                              ↓
                User approves → git commit + push → Netlify auto-deploys
```

## Onboarding endpoints

The manager exposes endpoints that automate the full form-to-live-site path. Confirmed from `docs/dealer-onboarding.md`:

| Endpoint | Purpose |
|---|---|
| `POST /api/onboarding/submission/:id/approve` | Approve a submitted dealer onboarding. Triggers scaffolding service: creates `sites/<slug>/`, writes `spoke.config.json`, copies platform-managed files, snapshots archetype to `_archetype.md`, runs `ui-ux-pro-max` skill to write `design.md`, symlinks `.claude/skills/ui-ux-pro-max`. |
| `POST /api/sites/<slug>/regenerate-design` | Idempotent. Re-runs the design-foundation skill against an existing dealer's `spoke.config.json` + `_archetype.md`. ~1-3 min (LLM call). Used when initial `design.md` failed or needs a fresh pass. |

The "paste-JSON modal" workflow described in the form repo README is the entry point for these endpoints — it's the same automated path, not a separate manual one.

## Dealer site stack (per the recipe)

Dealer spoke sites are intentionally simple:

- **Static HTML** with **Tailwind CDN** (no compile step)
- **Netlify Functions** for contact form submission (`/.netlify/functions/submit`)
- **Resend** for email notifications (API key in `data/secrets.json`, keyed by slug)
- Honeypot anti-spam: form has hidden `_gotcha` and `_redirect` fields
- No build command; publish dir is repo root

This is NOT Craft CMS. Craft is for the BGD-side custom client work and 2RM enterprise sites. The Terraplex spokes are static-by-design — fast to deploy, cheap to maintain, easy for Claude Code to edit.

## Hub versioning

The Terraplex hub is its own git repo (or submodule) at `hubs/terraplex/`. It's versioned via git tags (current: `v0.3.0`).

Each dealer's `spoke.config.json` pins to a specific `hub.version`. Hub content propagation (when Terraplex updates a product spec or rolls out required content across all dealers) is **Phase 3 work**, separate from per-dealer onboarding.

Pre-flight before any onboarding: hub must be on a clean tag matching origin. "Don't onboard against a mid-work hub."

## Slug rules (the slug is forever)

The canonical dealer slug touches **5 locations**:

1. `sites/<slug>/` directory name
2. `dealer.id` in `spoke.config.json`
3. `takenBy[].dealer` in `hubs/terraplex/archetypes/_index.json`
4. `data/sites.json` entry key
5. `data/secrets.json` entry key

Rules from the recipe:
- kebab-case, lowercase
- short and memorable, **not domain-derived** (`pyro-ag` not `pyroagllc-com`; `truss` not `truss-services-llc`)
- no special chars, underscores, or spaces
- unique across the network

Phase 2.75 was a slug rename exercise — the lesson is "the slug is forever." This is why the existing `trussservicesllc-com-com` entry is a real bug (domain-derived AND has a doubled `com`); fixing it touches all 5 locations.

## Hub structure (`hubs/terraplex/`)

```
hubs/terraplex/
├── spoke/
│   ├── spoke.schema.json    # JSON schema (AJV-validated against dealer configs)
│   └── questionnaire.json   # Source of truth for form fields + canonical service taxonomy
├── products/
│   ├── r-32.json + r-32.md  # Product specs + body copy (source of truth, hub-versioned)
│   └── i-19.json + i-19.md
├── archetypes/
│   ├── _index.json          # Catalog + uniqueness tracking (takenBy[])
│   ├── forge.md, tactical.md, terrain.md, clean.md, field.md, patriot.md
├── assets.json              # Hub-canonical asset references
└── design/
    └── propagation-v1.md    # Phase 3 plan: how hub updates roll out across dealers
```

Validation:
```bash
ajv validate -s hubs/terraplex/spoke/spoke.schema.json -d sites/<slug>/spoke.config.json
```

If the dealer config doesn't validate, fix the config. Schema changes are a separate phase.

## TODO policy in spoke configs

- **Required fields with no data**: literal string `"TODO"`. Schema allows it; flags real gaps.
- **Optional fields with no data**: omit entirely.

Keeps configs clean and makes "needs follow-up" visible.

## Tech stack

- **Backend:** Node.js + Express, ES modules
- **Frontend:** Vue 3 + Vite (SPA, no Nuxt)
- **Realtime:** Server-Sent Events for streaming Claude output
- **Auth:** Session-based with bcrypt (single-user now, multi-user planned for client access)
- **Process mgmt (prod):** PM2
- **Data:** JSON files on disk (`data/sites.json` registry, no DB yet)
- **Styling:** Custom CSS variables, dark theme, no Tailwind

## The skills approach

`.claude/skills/ui-ux-pro-max/` is committed to the manager repo and **symlinked into each new dealer repo at scaffold time**. At onboarding approval, the skill is invoked once per dealer to write `sites/<slug>/design.md` — the dealer's unique design foundation that complements the chosen archetype recipe.

Driver: `server/services/design-foundation.js`.
Recipe context: `docs/dealer-onboarding.md`.

This is a non-trivial pattern: skills become part of the productized deliverable, not just author-time tooling.

## Integrations

- **Claude Code (headless)** — spawned per chat message, scoped to one site directory at a time.
- **Netlify** — auto-deploys from git push; deploy status polled via Netlify API.
- **Refero** — see `docs/refero-integration.md`. Refero MCP tools (`refero_get_flow`, `refero_search_screens`, etc.) are exposed through claude.ai integration in this AIOS session, suggesting reference-image lookup is part of the design-foundation flow.
- **GitHub (deploy keys)** — for cloning private site repos.

## Security model

- Manager app is auth-gated (no public exposure).
- Allowed Claude Code tools: `Edit, Write, Bash, Read` (whitelist).
- Site path validation prevents traversal (`SITES_DIR` boundary).
- Rate limiting on `/api/chat`.
- Production runs behind Apache reverse proxy with HTTPS.
