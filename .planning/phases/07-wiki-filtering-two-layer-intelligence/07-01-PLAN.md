<objective>
Create the Layer 1 AIOS output filter that classifies content as operational vs. project-relevant before it reaches any wiki's raw/aios/ directory (WIKI-01). Then integrate the classifier into all three existing write surfaces so operational content is intercepted and never staged.

Purpose: This is the first intelligence layer in the two-layer filtering pipeline. Without it, every capture, chat drop, and session transcript flows unfiltered into project wikis regardless of whether the content is AIOS-internal (triage metadata, audit reports, system config) or genuinely project-relevant.

Output: A tested pure-function classifier module and updated write surfaces that gate on classification before calling writeRawDrop.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Operator input -> classifier | Content text from capture box or chat is untrusted user input processed by regex |
| Classifier -> writeRawDrop | Classification decision gates whether content reaches the wiki filesystem |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-01 | Tampering | classifyContent regex | accept | Classifier is advisory; false operational classification loses content to AIOS log but does not corrupt wiki. Operator can re-capture. Risk is low given routeContext always overrides for current surfaces. |
| T-07-02 | Denial of Service | classifyContent regex | accept | All regexes are simple alternation patterns with word boundaries. No backtracking risk. Input is operator-authored text, not external. |
| T-07-03 | Information Disclosure | content-classifier.ts | accept | Classifier runs server-side in Node.js. No client exposure. Content stays on local filesystem. |
