# Phase 6: Persistent To-Do List + Skill Lifecycle - Research

**Researched:** 2026-06-04
**Domain:** Claude Code skills (markdown-based), filesystem artifacts, AIOS data model
**Confidence:** HIGH

## Summary

Phase 6 delivers two independent but related capabilities: (1) a persistent, filesystem-based to-do list that survives across Claude Code sessions and supports both manual entries and future automated writes from triage (Phase 8), and (2) skill lifecycle changes -- retiring `/load-project` with a deprecation notice and rewiring `/onboard-client` to chain through the full intake-to-kickoff lifecycle.

This is a low-infrastructure phase. No new dependencies, no database, no API endpoints, no UI changes. All work is markdown skill files plus one new filesystem artifact (`todos/`). The existing AIOS UI in `aios-ui/` already has a Todo type system and ephemeral cache (`aios-ui/.aios-cache/todos-today.json`) from the v3 triage work. Phase 6's persistent to-do list is a separate, complementary artifact that lives in the claude-os repo root -- not a replacement for the UI's ephemeral triage cache.

**Primary recommendation:** Create `todos/` directory at claude-os repo root with `pending.md` as the single-file persistent store. Use a structured markdown format (task-list items with metadata in a parseable pattern) so both human operators and future automated consumers (Phase 8 triage, Phase 10 UI) can read and write it reliably.

### Pattern 1: Structured Markdown To-Do Format

**What:** A parseable but human-readable markdown format for persistent to-dos.
**When to use:** Any skill or automation that needs to write or read action items.

```markdown
# To-Do List

Items persist until explicitly completed. Completed items are moved to `completed.md`.

## Pending

- [ ] **Reply to Deann re: Thermal Kitchen launch date** `#follow-up`
  - Added: 2026-06-04
  - Source: triage
  - Client: kirk-financial / thermal-kitchen
  - Priority: high
  - Notes: Deann asked about the 6/16 launch timeline in her 6/3 email.

- [ ] **Review BrandOS dealer migration status** `#review`
  - Added: 2026-06-03
  - Source: manual
  - Priority: medium

- [ ] **Send AR email to Scott at Inside Out** `#email`
  - Added: 2026-06-02
  - Source: triage
  - Client: inside-out / inside-out-website
  - Priority: high
  - Notes: Draft sitting in Gmail. Content wrap done.
```

**Format rules:**
- Each item starts with `- [ ]` (pending) or `- [x]` (completed)
- Bold summary on the first line, optional hashtag category
- Indented metadata lines: `Added`, `Source`, `Client`, `Priority`, `Notes`
- `Source` is one of: `manual`, `triage`, `onboard`, `skill:<name>`
- `Client` uses the `client-slug / project-slug` compound form from clients.yaml
- `Priority` is one of: `high`, `medium`, `low`
- Items are appended at the bottom of the `## Pending` section
- Completed items get their checkbox marked `[x]` and are moved to `completed.md` periodically

### Pattern 2: Skill Deprecation via Redirect

**What:** Retire a skill by replacing its execution body with a deprecation notice while keeping the trigger phrases so Claude recognizes the command.
**When to use:** When a skill's functionality has been superseded by a better pattern.

```markdown
---
name: load-project
description: "DEPRECATED. Use when Justin says /load {project} or any variant. Prints a deprecation notice directing the operator to work directly in the project repo."
---

## This skill is deprecated

`/load-project` has been retired as of 2026-06-XX. Project wikis are now self-sufficient.

**What to do instead:**

1. Open the project repo directly: `cd ~/repos/<project-slug>/`
2. Claude Code will read the project's `CLAUDE.md` and `docs/wiki/WIKI-CLAUDE.md` automatically.
3. The wiki contains all decisions, state, and context that `/load-project` used to hydrate.

**Why this changed:** The llm-wiki pattern makes each project's wiki the canonical knowledge home. Loading context into the AIOS session is no longer necessary -- work happens in project repos, not from claude-os.

If you need cross-project status, use `/weekly-project-status` instead.
```

### Pattern 3: Skill Chaining via Sequential Execution

**What:** One skill invokes another by instructing Claude to run a subsequent skill command after completing its own steps.
**When to use:** When a skill's workflow naturally leads into another skill's domain.

The `/onboard-client` skill currently ends at Phase 6 (Close) with a summary. The rewired version adds three new phases:

```
Phase 5 (Write) -- existing writes complete
    |
    v
Phase 6 (Kickoff) -- NEW
    "Now run /kickoff-project {project-slug}"
    Claude follows the kickoff-project skill instructions
    |
    v
Phase 7 (Initial STATE.md) -- NEW
    Write initial STATE.md using the state-prompt.md format
    Copy to state/<slug>.md
    |
    v
Phase 8 (Final Close) -- MODIFIED
    Updated summary showing full lifecycle completion
    Decisions log entry updated to reflect kickoff + STATE.md
```

### Anti-Patterns to Avoid

- **Storing email threads in the to-do list.** TODO-04 is explicit: email threads are ephemeral triage output, not persistent actions. Only extracted action items (decisions, follow-ups, tasks) go into `todos/pending.md`. The thread itself stays in `aios-ui/.aios-cache/todos-today.json`.

- **Building a task management system.** This is a persistent scratch pad, not Jira. No IDs, no state machines, no assignments, no sprint tracking. The operator adds items, completes them, and moves on. Keep it dead simple.

- **Auto-generating to-dos from every session.** The to-do list is for explicit action items, not a session log. The Phase 5 STATE.md hook handles session-level state capture. To-dos are things Justin needs to *do*, not things he *did*.

- **Making `/onboard-client` brittle by requiring `/kickoff-project` to succeed.** The chained flow should be graceful: if kickoff fails (e.g., repo doesn't exist, user cancels the interview), onboard-client should still complete its core writes (clients.yaml, memory, decisions log) and report that kickoff needs to be run manually.

- **Overwriting the existing `/load-project` SKILL.md silently.** The deprecation notice must preserve the trigger phrases in the frontmatter `description` field so Claude still recognizes `/load {project}` and can explain the deprecation. If the description is deleted, the skill becomes invisible.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Unique IDs for to-do items | UUID generator, counter, database | Let items be identified by their summary text + added date | At this scale (<100 items), text matching is sufficient. Phase 10 UI can assign stable IDs when it reads the file. |
| STATE.md format | New template from scratch | Existing `scripts/state-hook/state-prompt.md` format | Consistency with Phase 5's auto-generated STATE.md. The initial STATE.md from onboard should look identical to the hook-generated ones. |
| Project resolution from slug | New lookup logic | Existing `clients.yaml` parsing pattern used by `state-session-end.js` and `/load-project` | Every skill that resolves slugs uses the same YAML structure. Don't invent a new resolution algorithm. |
| Wiki detection | Custom wiki-finding code | Existing `detectWikiRoot()` pattern from `state-session-end.js` | `/kickoff-project` creates the wiki; the SESSION_END hook's `detectWikiRoot()` is the canonical pattern for finding it afterward. |

**Key insight:** This phase creates no new infrastructure. It composes existing patterns (skill markdown, filesystem artifacts, YAML registry) into two new behaviors (persistent to-do, lifecycle chain). The value is in the wiring, not the plumbing.

## Common Pitfalls

### Pitfall 1: To-Do Format Drift Between Manual and Automated Writes

**What goes wrong:** The operator adds items in freeform prose. Phase 8 triage adds items in a structured format. Phase 10 UI tries to parse the file and breaks on the freeform entries.
**Why it happens:** No format specification, or the specification isn't enforced.
**How to avoid:** Define the format clearly in the research (done above). The skill instructions for any writer (manual add, triage, onboard) must include the format template. The format must be simple enough that hand-editing doesn't require memorizing a schema.
**Warning signs:** Items without the `Added:` or `Source:` metadata lines. Items not starting with `- [ ]`.

### Pitfall 2: `/onboard-client` Chain Breaks Mid-Flow

**What goes wrong:** The `/kickoff-project` step fails (repo doesn't exist, user declines the interview, filesystem error) and the operator is left with a half-onboarded client.
**Why it happens:** The chain assumes linear success. Real flows have branching failure modes.
**How to avoid:** Make the chain graceful: core onboard writes (clients.yaml, memory, reference doc, decisions log) complete BEFORE the kickoff chain starts. If kickoff fails, the operator has a usable client entry and can run `/kickoff-project` manually later.
**Warning signs:** "Something went wrong during kickoff" with no clients.yaml entry to show for it.

### Pitfall 3: Deprecation Notice Breaks `/load` Trigger Recognition

**What goes wrong:** After editing the SKILL.md, Claude no longer recognizes "/load brandos" as a command because the `description` field in the frontmatter was changed or removed.
**Why it happens:** Claude Code matches user input against the `description` field's trigger phrases.
**How to avoid:** Keep ALL existing trigger phrases in the frontmatter `description` field. Only change the execution body below the frontmatter separator.
**Warning signs:** User says "/load brandos" and Claude doesn't know what to do.

### Pitfall 4: Initial STATE.md Doesn't Match Hook-Generated Format

**What goes wrong:** The initial STATE.md written by `/onboard-client` has a different structure than the one produced by `state-session-end.js` + `state-prompt.md`. Phase 10 UI's parser handles one format but not the other.
**Why it happens:** Two different code paths produce the same artifact without sharing a template.
**How to avoid:** Use the exact format from `scripts/state-hook/state-prompt.md`. The initial STATE.md should have the same H1, metadata line, and H2 sections. Content will differ (it's a seed, not a session summary) but the structure must match.
**Warning signs:** `state/<slug>.md` files with inconsistent heading structures.

### Pitfall 5: Completed Items Accumulate in pending.md

**What goes wrong:** Over weeks, `pending.md` grows with dozens of `[x]` completed items, making it hard to scan for open work.
**Why it happens:** No archival mechanism.
**How to avoid:** Include a `completed.md` file in the `todos/` directory. Periodically (or as part of a future routine), move `[x]` items from `pending.md` to `completed.md`. This can be manual for Phase 6 and automated in a later phase.
**Warning signs:** `pending.md` is more than 50% completed items.
