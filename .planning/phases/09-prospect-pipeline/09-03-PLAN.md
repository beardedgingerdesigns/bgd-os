<objective>
Modify the `/onboard-client` skill to detect and leverage existing prospect docs. When onboarding a client that has a prospect doc at `prospects/<slug>.md`, the skill reads the accumulated intelligence (Timeline, Problem Map, Notes, contact map) and pre-fills answers to interview questions. The operator sees only the gaps -- questions the prospect doc can't answer -- producing a shorter, more targeted interview.

This is what makes the prospect pipeline pay off at conversion time. Months of accumulated email intel collapse into a few focused questions instead of a cold-start interview.
</objective>

    Step 0 -- Prospect Context Detection:
    1. If the user provided a slug (e.g., `/onboard-client revolution`), check if `prospects/<slug>.md` exists
    2. If the user said "new client" and provided a slug in Phase 1 Q1a, check after Q1a
    3. If a prospect doc is found:
       a. Read the full prospect doc
       b. Parse YAML frontmatter (contacts map, first_seen, source)
       c. Parse body sections: Timeline (extract key intel from most recent 5 entries), Problem Map (all entries), Notes (all entries), Pricing Strategy (if any content)
       d. Build a `prospect_context` object with extracted answers:
          - `display_name`: from doc H1 heading
          - `contacts`: from frontmatter contacts map
          - `bucket`: pre-fill as "active" (D-12: conversion sets active, NOT paying. Paying requires first invoice.)
          - `problem_summary`: from Problem Map section
          - `engagement_history`: from Timeline section (summarized)
          - `pricing_context`: from Pricing Strategy section
          - `source`: from frontmatter source field
       e. Display to user:
          ```
          ✓ Found prospect doc: prospects/<slug>.md
            First seen: <first_seen>
            Source: <source>
            Contacts: <count> tracked
            Timeline entries: <count>

          Pre-filling from prospect intelligence. You'll confirm each answer.
          ```
    4. If no prospect doc found: proceed with normal onboard flow unchanged

    Then modify Phases 1-3 to use prospect_context:

    Phase 1 modifications:
    - Q1b (Display name): pre-fill from prospect_context.display_name, ask for confirmation
    - Q1c (Bucket): pre-fill as "active" with note "Converting from prospect — set to 'paying' after first invoice" (per D-12: active means signed but not yet invoicing)

    Phase 2 modifications:
    - Project scope question: pre-fill from prospect_context.problem_summary, ask "Based on prospect intel, here's what they need: [summary]. Correct, or want to adjust?"
    - Any question about the client's business/industry: extract from Timeline intel, pre-fill

    Phase 3 modifications:
    - Contact info: pre-fill from prospect_context.contacts (all tracked contacts with roles)
    - Source tracking: note "Originally sourced as prospect via: <source>"

    Key rule: NEVER silently assume a pre-filled answer is correct. Always show the pre-filled value and ask for confirmation or adjustment. The prospect doc is intelligence, not gospel.

    For questions with no prospect doc coverage: ask normally (these are the "gaps" that make the interview shorter but still complete).
