    - description: Use when Justin says "register prospect X", "new prospect", "add X as a prospect", "track this lead", or any variant of prospect registration. Interactive intake creates prospect doc and clients.yaml entry.

    Execution flow (3 steps):

    STEP 1 -- Gather prospect identity via AskUserQuestion:
    - Q1: Prospect slug (free-text, kebab-case, validate uniqueness against clients.yaml AND existing prospects/ files)
    - Q2: Display name (free-text, human-readable company/person name)
    - Q3: Source (AskUserQuestion: "Email thread", "Referral from [name]", "Cold outreach", "Other")
    - Q4: Primary contact name (free-text)
    - Q5: Primary contact email (free-text)
    - Q6: Initial notes (free-text, optional -- any context Justin has about the prospect right now)

    STEP 2 -- Scaffold prospect doc at `prospects/<slug>.md`:

    Frontmatter (D-03):
    ```yaml
    ---
    status: active
    status_changed_at: <today>
    previous_status: null
    contacts:
      <email>:
        name: <name>
        email: <email>
        role: primary
        last_interaction: <today>
    first_seen: <today>
    last_activity: <today>
    source: <source from Q3>
    ---
    ```

    Body sections (D-04):
    ```markdown
    # <Display Name>

    ## Timeline
    <!-- Reverse-chrono: newest at top (D-05) -->

    ### <today>
    - Registered as prospect. Source: <source>.
    <initial notes if provided>

    ## Pricing Strategy
    <!-- Working surface for AIOS pricing conversations -->

    ## Problem Map
    <!-- Business problems the prospect has mentioned -->

    ## Notes
    <!-- Freeform manual observations -->

    ## Next Steps
    <!-- What needs to happen to advance the deal -->
    ```

    STEP 3 -- Add entry to clients.yaml:
    Read clients.yaml. Append a new entry. CRITICAL: contacts in clients.yaml are simple email strings (e.g., `person@company.com` or `@company.com`), NOT objects. They currently live under `projects[].contacts`. Since new prospects have no projects yet, add a root-level `contacts:` field with the same simple email string format:
    ```yaml
    - slug: <slug>
      name: <display_name>
      bucket: prospects
      contacts:
        - <primary_contact_email>
      projects: []
    ```

    IMPORTANT: The root-level `contacts:` array uses the SAME format as `projects[].contacts` — simple email strings and @domain patterns. This is NOT the same as the prospect doc's frontmatter contacts map (which has name/email/role objects for the dossier). Two separate surfaces:
    - clients.yaml contacts = simple emails for triage matching
    - prospect doc frontmatter contacts = rich objects for the dossier

    Triage matching (Plan 02) will be extended to check root-level contacts in addition to project-level contacts. This is how prospects without projects get matched.

    Validate the YAML parses correctly after the edit. If parse fails, revert and report error.

    After successful registration, output:
    ```
    ✓ Prospect registered: <display_name>
      Doc: prospects/<slug>.md
      Entry: clients.yaml (bucket: prospects)

    Triage will now match emails from <email> to this prospect.
    ```
