    Create `.claude/skills/convert-prospect/SKILL.md` as a user-level skill.

    Skill frontmatter:
    - name: convert-prospect
    - description: Use when Justin says "convert prospect X", "X is now a client", "close the deal on X", "onboard X from prospect", or any variant of converting a prospect to an active client. Reads prospect dossier, flips bucket, archives doc, seeds wiki, chains to /onboard-client.

    Execution flow (6 steps, two-phase commit pattern):

    PHASE A: PREPARE (reversible — nothing permanent until Phase B)

    STEP 1 -- Validate prospect:
    1. Accept slug as argument: `/convert-prospect <slug>`
    2. Check `prospects/<slug>.md` exists. If not: error "No prospect doc found at prospects/<slug>.md. Check the slug or register first with /register-prospect."
    3. Read the full prospect doc (frontmatter + body)
    4. Check frontmatter status:
       - If `converted`: error "Prospect <slug> was already converted. See prospects/converted/<slug>.md"
       - If `converting`: this is a RETRY after a failed conversion. Check state:
         a. If wiki exists AND STATE.md exists AND prospect seed files exist in wiki raw/aios/ (prospect-timeline-*, prospect-problems-*): skip to Phase B (finalize). All Phase A work completed.
         b. If wiki exists AND STATE.md exists BUT seed files missing: resume from Step 4 (seed wiki). Onboard/kickoff completed but seeding failed.
         c. If wiki does NOT exist: resume from Step 3 (chain to onboard-client). The status is already `converting` (race-safe).
         d. Display: "Resuming conversion of <slug> from previous attempt..."
       - If `active`, `warm`, or `cold`: proceed normally from Step 2.
    5. Display prospect summary for confirmation:
       ```
       ╔══════════════════════════════════════════════════════════════╗
       ║  CONVERTING PROSPECT                                        ║
       ╚══════════════════════════════════════════════════════════════╝

       Prospect: <display_name>
       First seen: <first_seen>
       Source: <source>
       Contacts: <count>
       Timeline entries: <count>
       Status: <current_status>

       This will:
       1. Mark prospect as 'converting' (prevents triage race)
       2. Run /onboard-client → /kickoff-project with pre-filled context
       3. Seed new project wiki with prospect intelligence
       4. Flip bucket in clients.yaml from 'prospects' to 'active'
       5. Archive prospect doc to prospects/converted/

       ──────────────────────────────────────────────────────────────
       → Continue with conversion?
       ──────────────────────────────────────────────────────────────
       ```
    6. Use AskUserQuestion: "Convert <display_name> to active client?" with options "Yes, convert" and "Cancel"

    STEP 2 -- Mark converting (prevents triage race condition):
    1. Update prospect doc frontmatter: set `status: converting`
    2. This prevents triage from dispatching to a doc mid-conversion. Triage sees status: converting and defers dispatch (retried after conversion completes).
    3. The prospect doc remains at `prospects/<slug>.md` throughout the conversion — NOT archived yet.
    4. IMPORTANT: Do NOT flip the clients.yaml bucket yet. Keep `bucket: prospects` during conversion. Triage matching with bucket: prospects + status: converting = deferred dispatch (no lost intel). If we flipped to active now, triage would try wiki dispatch with no project/wiki and silently fail.

    STEP 3 -- Chain to onboard-client (conversion mode):
    1. Invoke /onboard-client with the slug. Since the prospect doc is STILL at `prospects/<slug>.md` (not yet archived) and bucket is still `prospects`, onboard-client Step 0 will find it and pre-fill from it.
    2. IMPORTANT: onboard-client must NOT create a duplicate clients.yaml entry. The entry already exists (bucket is still `prospects`). Onboard-client must detect the existing entry and UPDATE it (add project, update contacts) instead of appending a new one. This is the "conversion mode" behavior — triggered by detecting `status: converting` in the prospect doc frontmatter.
    3. /onboard-client will chain to /kickoff-project which creates the wiki
    4. If /onboard-client or /kickoff-project fails: prospect doc still at original location, bucket still `prospects`, status `converting`. Recovery: reset status to `active` in prospect doc and retry /convert-prospect. The bucket was never flipped, so no inconsistent state.

    STEP 4 -- Seed wiki with prospect knowledge:
    1. After /kickoff-project completes and the wiki exists, extract from the prospect doc:
       - Timeline section content → stage as `{wiki}/raw/aios/prospect-timeline-<slug>.md` with frontmatter `source: prospect-conversion, type: timeline`
       - Problem Map section content → stage as `{wiki}/raw/aios/prospect-problems-<slug>.md` with frontmatter `source: prospect-conversion, type: problem-map`
       - Pricing Strategy section content (if non-empty) → stage as `{wiki}/raw/aios/prospect-pricing-<slug>.md` with frontmatter `source: prospect-conversion, type: pricing-strategy`
    2. These files are staged into the wiki raw/aios/ directory per the staged ingestion convention (ADR-0004). The wiki's ingest pipeline will curate them.
    3. STATE.md creation and sync is handled by /onboard-client → /kickoff-project chain (single owner per PROSPECT-05). After /kickoff-project completes, verify STATE.md exists at `state/<slug>.md`. If MISSING: this is a BLOCKING failure. Do NOT proceed to Phase B (archive). Display error: "STATE.md not found at state/<slug>.md after onboard/kickoff chain completed. Conversion cannot finalize. The prospect doc remains at prospects/<slug>.md with status: converting. Fix: manually verify /kickoff-project output, then re-run /convert-prospect <slug>." This ensures PROSPECT-05 is satisfied — STATE.md MUST exist before conversion is considered complete.

    PHASE B: FINALIZE (only after all Phase A steps succeed)

    STEP 5 -- Flip bucket in clients.yaml (deferred from Phase A):
    1. Read clients.yaml
    2. Find the entry with matching slug
    3. Change `bucket: prospects` to `bucket: active` (D-12 -- NOT paying. Active means signed but not yet invoicing.)
    4. Validate YAML parses correctly after edit
    5. Write updated clients.yaml
    6. This happens AFTER wiki/onboard/STATE.md all succeed. The delayed bucket flip ensures triage never sees `bucket: active` with no wiki to dispatch to.

    STEP 6 -- Archive prospect doc (point of no return):
    1. Copy `prospects/<slug>.md` to `prospects/converted/<slug>.md`
    2. Update the archived copy's frontmatter: set `status: converted`, add `converted_date: <today>`
    3. Delete the original `prospects/<slug>.md`
    4. Archive + bucket flip are the LAST steps — only after wiki exists, STATE.md is synced, and onboard completed. If any Phase A step failed, the doc stays at its original location with bucket still `prospects` and status `converting`.

    Final output:
    ```
    ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
     PROSPECT CONVERTED ✓
    ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

    <display_name> is now an active client.

    ✓ Bucket: prospects → active
    ✓ Archived: prospects/converted/<slug>.md
    ✓ Wiki seeded: <wiki_path>/raw/aios/ (3 files staged)
    ✓ Onboarded via /onboard-client
    ✓ STATE.md synced to state/<slug>.md

    Next: bucket moves to 'paying' when first invoice is sent.
    ```
