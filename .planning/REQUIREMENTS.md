# Requirements: claude-os v2.0 — AIOS v2 Dispatcher + Strategic Partner

## Milestone Requirements

### State Management (STATE)
- [x] **STATE-01**: End-of-session hook generates STATE.md in project wiki with status, accomplishments, next steps, blockers, and key dates
- [x] **STATE-02**: End-of-session hook syncs STATE.md copy to claude-os `state/<slug>.md`
- [x] **STATE-03**: Hook is gated on session substance (skips trivial sessions based on edits/messages/commits threshold)
- [x] **STATE-04**: LLM summarizes session context into tight STATE.md when substance threshold is met

### Triage (TRIAGE)
- [x] **TRIAGE-01**: Triage runs as a scheduled routine (~2hr intervals during waking hours), not a manually triggered skill
- [x] **TRIAGE-02**: Triage output includes ranked email threads with inline draft replies ready for revision
- [x] **TRIAGE-03**: Triage extracts persistent action items (to-do list) from email content
- [x] **TRIAGE-04**: Triage auto-generates dispatch handoffs to project wikis when emails contain project-relevant intelligence
- [x] **TRIAGE-05**: Dispatch filter uses heuristic categories first, LLM tiebreaker for ambiguous cases

### Wiki Filtering (WIKI)
- [x] **WIKI-01**: AIOS output filter classifies content as operational vs. project-relevant before staging to `raw/aios/`
- [x] **WIKI-02**: Wiki ingest evaluates incoming `raw/aios/` files against current wiki knowledge
- [x] **WIKI-03**: Wiki ingest produces three outcomes: promote (new info), skip (redundant), flag (contradicts existing)
- [x] **WIKI-04**: Flagged contradictions surface to operator for resolution rather than auto-overwriting

### Prospect Pipeline (PROSPECT)
- [ ] **PROSPECT-01**: Prospect doc structure at `prospects/<slug>.md` for knowledge accumulation
- [ ] **PROSPECT-02**: Triage tags relevant email threads to prospects via explicit contacts in `clients.yaml`
- [ ] **PROSPECT-03**: `/onboard-client` reads prospect doc and asks targeted questions based on accumulated knowledge
- [ ] **PROSPECT-04**: Conversion trigger flips bucket, runs `/kickoff-project` seeded with prospect knowledge + interview answers
- [ ] **PROSPECT-05**: Initial STATE.md created in new wiki and synced to claude-os at conversion

### To-Do List (TODO)
- [x] **TODO-01**: Persistent to-do list stored as first-class AIOS artifact in claude-os
- [x] **TODO-02**: Action items generated by triage persist until completed
- [x] **TODO-03**: Operator can manually add action items
- [x] **TODO-04**: Email threads managed by triage are ephemeral (not stored in to-do list)

### AIOS UI (UI)
- [ ] **UI-01**: Dashboard shows triage queue, to-do list, project status cards, MRR number, and prospect cards
- [ ] **UI-02**: Triage surface supports inline email reply revision and sending
- [ ] **UI-03**: Strategic chat as embedded terminal for cross-cutting business partner conversations
- [ ] **UI-04**: Project cards are read-only status displays powered by last-known-state
- [ ] **UI-05**: Clicking a project card opens it externally (VS Code or new Claude terminal session)
- [ ] **UI-06**: Admin section for skill launchers, research archives, and settings
- [ ] **UI-07**: Retire per-project chat, project brief caching, and per-project Communications section

### Skill Lifecycle (SKILL)
- [x] **SKILL-01**: Retire `/load-project` skill
- [x] **SKILL-02**: `/onboard-client` triggers full lifecycle: intake → `clients.yaml` → `/kickoff-project` → STATE.md → decisions log

## Future Requirements (deferred from scoping)

- v1.0 deferred items (CAPX, RITX, ADMC, BRDO) — evaluate relevance under the new dispatcher model; some may be obsoleted by the reframe
- Proactive pattern scanning routine (cross-project signal detection)
- Embedded terminal cost optimization for Claude pricing changes

## Out of Scope

- Multi-user / auth / cloud deploy — locked to local-only single-operator per ADR 0001 §1
- Database / persistent cache — filesystem is source of truth per ADR 0001 §4
- Project-level work inside AIOS — operator leaves AIOS to work in project repos
- Full email client — AIOS is a triage + reply layer, not an inbox replacement
- Calendar management — calendar is read-only awareness, not a scheduling tool
